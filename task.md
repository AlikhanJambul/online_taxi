Техническое Задание (ТЗ): Сервис заказа поездок (Ride-Hailing App)
1. Общее описание проекта
   Разработка программного комплекса для заказа поездок, объединяющего мобильные приложения для пользователей и серверную часть с микросервисной архитектурой. Система обеспечивает поиск ближайших водителей, расчет стоимости, стриминг геолокации в реальном времени и управление жизненным циклом поездки.

2. Ролевая модель
   Система поддерживает три типа пользователей:

Пассажир (Passenger): Регистрация/авторизация, управление профилем (аватар, email), поиск такси, отслеживание авто на карте, оплата поездки, выставление оценки водителю.

Водитель (Driver): Регистрация/авторизация, профиль автомобиля (марка, цвет, госномер), выход на линию, получение предложений о поездке, стриминг своих координат, изменение статуса поездки, оценка пассажира.

Администратор (Admin): Авторизация (без регистрации), просмотр аналитики (дашборд), управление пользователями (одобрение профилей водителей, блокировка/разблокировка), просмотр истории поездок и маршрутов.

3. Технологический стек
   Frontend: Flutter (iOS, Android).

Backend: Go (Golang).

Архитектура API: gRPC (с использованием Bidirectional и client Streaming).

Основная База Данных: PostgreSQL (хранение пользователей, поездок, профилей, отзывов).

In-Memory Кэш & Геолокация: Redis (GEO-индексы для поиска машин в радиусе, хранение текущих координат).

Брокер сообщений: RabbitMQ (асинхронные события: уведомления, мэтчинг, биллинг).

Файловое хранилище: MinIO / AWS S3 (хранение пользовательских аватаров).

Инфраструктура: Docker, Docker Compose.

4. Архитектура микросервисов и Функционал
   4.1. Auth & User Service (Сервис авторизации и пользователей)
   Регистрация и аутентификация по номеру телефона и паролю.

Генерация JWT (Access Token на 15 мин, Refresh Token на 30 дней).

Поддержка механизма Refresh Token Rotation и мультустройств (сессии привязаны к device_id).

Загрузка и обновление аватарок через MinIO.

Регистрация данных об автомобиле для водителя (ожидание аппрува от админа).

4.2. Location & Matching Service (Сервис геолокации и мэтчинга)
Стриминг (client-to-Server): Водитель отправляет координаты каждые 3-5 секунд по gRPC.

GEO-Индексация: Координаты обновляются в Redis.

Радар: При создании заказа сервис ищет топ-N свободных водителей в радиусе 3 км от пассажира.

Стриминг (Bidi Streaming): Отправка координат водителя на устройство пассажира во время ожидания и поездки.

4.3. Trip Service (Сервис поездок)
Управляет бизнес-логикой и машиной состояний (State Machine) заказа.

Калькулятор: Предварительный расчет стоимости поездки (Базовая ставка + Дистанция * Тариф).

Жизненный цикл: SEARCHING -> ACCEPTED -> ARRIVED -> IN_PROGRESS -> COMPLETED / CANCELLED.

Запись маршрута: Сбор массива координат из Redis и пакетное сохранение исторического трека поездки в базу данных после её завершения.

4.4. Notification Service (Сервис уведомлений)
Внутренний консьюмер (слушатель) очередей RabbitMQ.

Отправка Push-уведомлений при изменении статуса (например: "Водитель ожидает").

Запрос оценки (рейтинга) после завершения поездки.

4.5. Admin Service (Панель управления)
Мониторинг активных сессий и поездок в реальном времени.

Управление статусами водителей (PENDING -> APPROVED).

Блокировка злонамеренных пользователей (is_blocked = true).

5. Структура Баз Данных (PostgreSQL)
   Система использует единую реляционную базу со следующими основными таблицами:

users: Общая таблица пользователей (id, телефон, хэш пароля, роль, рейтинг, статус блокировки).

sessions: Управление Refresh-токенами (user_id, refresh_token, device_id, expires_at) для защиты сессий.

driver_profiles: Данные автомобилей, привязанные к users связью 1-к-1.

trips: История и текущие заказы (id, пассажир, водитель, координаты А и Б, цена, статусы, таймстемпы).

reviews: Оценки и комментарии по итогам поездок (влияют на перерасчет рейтинга в таблице users).

6. Топология RabbitMQ (Event-Driven Architecture)
   Для обеспечения слабой связности микросервисов используется паттерн Publish/Subscribe через обменники типа Topic.

Exchange trip.events:

trip.created -> очередь matching.find_driver (поиск авто).

trip.status.changed -> очередь notification.push (информирование юзера).

trip.completed -> очереди billing.process_payment (списание) и rating.ask_review (запрос оценки).

Exchange user.events:

user.driver_approved -> очередь notification.push.

Exchange location.events:

location.trip_track -> очередь history.save_track (сохранение гео-трека после поездки).

7. План и этапы разработки
   Этап 1: Инфраструктура и Контракты. Поднятие Docker-контейнеров (Postgres, Redis, RMQ, MinIO). Написание .proto файлов (API-First подход).

Этап 2: Ядро аутентификации (Auth Service). Реализация регистрации, JWT, хэширования (bcrypt), работы с сессиями.

Этап 3: Создание поездок (Trip Service). CRUD операции для поездок, машина состояний заказа. Интеграция с RabbitMQ.

Этап 4: Реалтайм геолокация. Настройка Redis GEO, реализация gRPC стриминга координат. Мэтчинг пассажира и водителя.

Этап 5: Frontend (Flutter). Реализация мобильных интерфейсов, подключение карт, обработка gRPC-ответов и стримов.

Этап 6: Админ-панель. Реализация методов просмотра статистики и управления доступом.
